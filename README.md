# GhostPort 
### Cloud-Native Distributed Deception & Automated Alerting Platform

> **⚠️ Archival Notice:** This project is a completed architectural Proof of Concept (PoC) and is currently archived. Active development has been shifted to **AccessGraph** to focus on advanced multi-cloud identity attack path scaling at an enterprise level.

GhostPort is an automated, high-leverage SaaS/PaaS platform designed to deploy lightweight, invisible digital tripwires (honey-tokens, decoy configurations, and listener endpoints) seamlessly across distributed infrastructure, repositories, and internal networks. 

Traditional honeypots are isolated, brittle, and require intrusive operating system modifications. GhostPort resolves this by using a high-throughput, split-plane architecture. When a malicious actor breaches a perimeter and interacts with a decoy asset, GhostPort silently intercepts the interaction, fingerprints the adversary, and routes low-latency telemetry to a centralized control dashboard—completely bypassing detection by the attacker.

## 🏗️ System Architecture

GhostPort operates on a decoupled **split-plane design** to maintain defensive invisibility and survive high-velocity network scanning attacks.

###  Data Flow & Workflows
1. **Deception Ingestion:** Cryptographically signed tokens (**GhostTokens**) are generated by the Control Plane backend and mapped to specific tenant IDs.
2. **Telemetry Triangulation:** The distributed edge agent sniffs network layer data (`Source IP`, `User-Agent`, `Layer 7 headers`, `TLS fingerprints`, and raw payload strings) immediately upon asset interaction.
3. **Asynchronous Stream Queueing:** To prevent database connection pool exhaustion during a brute-force scan, event telemetry is dropped directly into a high-speed asynchronous stream engine (Redis Streams/Kafka).
4. **Enrichment & Notification:** Worker engines ingest events from the stream queue concurrently, enrich the data with GeoIP/ISP variables, persist the state to PostgreSQL, and dispatch real-time JSON alert packages via client webhook pipelines within milliseconds of initial contact.

##  Tech Stack & Microservice Routing

* **Frontend Web Console:** Next.js (React), Tailwind CSS, Lucide Icons, and Three.js / React Three Fiber for rendering real-time 3D attack node relationship graphs.
* **Backend Control Plane:** Go (Golang) / Node.js utilizing Fastify and Gin frameworks for high-throughput, low-overhead REST and gRPC endpoints.
* **Database & Caching Layer:** PostgreSQL (via Supabase) with native **Row-Level Security (RLS)** ensuring multi-tenant isolation; Redis handles session persistence, rate limiting, and ingestion streams.
* **Edge Core Tooling Engine:** Native multi-threaded network socket abstractions combined with `libpcap`/`WinPcap` bindings for raw packet sniffing and flag interception.
* **Deployment & Infrastructure:** Containerized via Docker, deployed via AWS Fargate/ECR for serverless scalability, and driven by GitHub Actions for continuous integration.

##  Mitigating Physical & Architectural Bottlenecks

### 1. Mass Scanning Ingestion Failure (The Scanner Flood)
* **The Issue:** Running multi-threaded network tools like `Nmap -T4` or `Masscan` against our edge honey-listeners spikes ingestion to thousands of events/sec, which would instantly throttle or drop direct database connection pools.
* **The Fix:** Engineered an ultra-fast Redis Stream buffer in front of the microservices, decoupling real-time network packet collection completely from storage engine writes.

### 2. Defensive Leak Prevention (Telemetry Timeouts)
* **The Issue:** If the central cloud ingestion endpoint is down, an edge agent waiting to resolve a network write will hang, creating observable latency that tips off an attacker.
* **The Fix:** Built a background, non-blocking asynchronous socket daemon on the edge that buffers events locally in a memory ring-buffer and retries backoffs without throwing explicit application-layer errors.

### 3. Distributed Database Deadlocks (State Tracking Race Conditions)
* **The Issue:** When tracking compound attack chains across multiple decoys concurrently, concurrent database workers attempting mutations on the exact same incident row caused database deadlocks.
* **The Fix:** Implemented a distributed locking scheme using Redis (`Redlock`) and refactored the persistence layer to follow an incremental, append-only log strategy rather than instant row modifications.
